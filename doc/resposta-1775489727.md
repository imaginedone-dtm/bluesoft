Resumo

Deploy de todas as customizações implementadas na org de desenvolvimento para o ambiente de homologação. Inclui modelo de dados (56 campos custom), 2 pipelines de venda (Acelerato + ERP) com Record Types, Paths e Layouts dedicados, hierarquia de segurança (Roles + Permission Sets), e componentes de UI.

Pré-Deploy (Checklist)

1. Conectar a org de homologação
[7:24 PM]# Autenticar na org homolog
sf org login web --alias BluesoftHML --instance-url https://bluesoft--hml.sandbox.my.salesforce.com

# Confirmar conexão
sf org display --target-org BluesoftHML




2. Inventariar o que já existe em homolog

# Verificar campos custom existentes (evitar conflito)
sf sobject describe -s Account --target-org BluesoftHML | grep "__c"
sf sobject describe -s Lead --target-org BluesoftHML | grep "__c"
sf sobject describe -s Opportunity --target-org BluesoftHML | grep "__c"

# Verificar Record Types existentes
sf data query -q "SELECT DeveloperName, IsActive FROM RecordType WHERE SobjectType='Opportunity'" --target-org BluesoftHML

# Verificar Roles existentes
sf data query -q "SELECT DeveloperName FROM UserRole" --target-org BluesoftHML

# Verificar Permission Sets
sf data query -q "SELECT Name FROM PermissionSet WHERE IsCustom=true" --target-org BluesoftHML




3. Validação local (dry-run)

# Validar o pacote SEM deployar
sf project deploy start --manifest manifest/package.xml --dry-run --target-org BluesoftHML




Se o dry-run falhar, analisar erros antes de prosseguir.

Alternativa: Deploy único via manifest

Se preferir um deploy único (não por ondas), usar o manifest:

# Validar primeiro (dry-run)
sf project deploy start --manifest manifest/package.xml --dry-run --target-org BluesoftHML

# Deployar
sf project deploy start --manifest manifest/package.xml --target-org BluesoftHML




:warning: O deploy por manifest respeita a ordem definida no package.xml, mas em caso de falha por dependência, voltar para o deploy por ondas.
